Data Lifecycle Checklist

 

This checklist is to be used to ensure that all aspects of the Data Lifecycle have been properly documented and implemented for client projects. (Ingested, Transformed, Curated, Queried, Explored, Analyzed and visualized, Stored)

 

In general, all of this information should go into the Project Design Document either as specific sections or as links to other documents as appropriate. 

 

Ingestion
  • Source to target mapping (What are all of the sources? Where are they going? Ideally, we'd have this down to the column level but at a minimum we should have this at the app/database level)
  • Source documentation
    • Constraints
    • Update frequency/timing
    • Links to any existing client documentation

 

Transformation
  • Business rule definitions (not every column has rules, but every rule needs to be documented)
  • Tool documentation (when, where, and how are these transformations done?)
  • Transformation rules
    • Standard transformations (for example, all Datetimes get converted to UTC)
    • Specific column transformations
  • Unit tests
    • Should be documented using the Test Design Document
    • What tools are being used?
    • Are the tests Automated? How?
    • How are the results shared with relevant stakeholders?

 

Curation
  • Data Model
    • Entity relationships
    • How does this project's data fit into the overall data ecosystem?
    • Who has stewardship of the Model once the project is complete?
  • Data Governance
    • Are all major business rule definitions the same across units? (For example, do Sales and Finance agree on the definition of terms used by both?)
    • Who has stewardship of the Rules and Transformations once the project is complete?
    • Is there a tool being used? [Should have link to DG tool matrix]
  • Lineage
    • Is there clear mapping of columns from source to analytics?
    • If a change is made upstream, will it properly propagate downstream?
    • Is there a tool being used? 
    • Is the lineage easily searched by end users?

 

Data Quality Checks
  • Cleanliness (Null counts, Orphan records, unmatched lookup values, etc)
  • Completeness checks (Did all of the data from the source get pulled? If a backfill happens, can it be detected?)
  • Appropriateness checks (bounding, negative values, unicode characters, etc)
  • Tools/Automation
    • Is a tool being used?
    • How are results conveyed to users?
    • How are problems detected and fixed in a timely fashion?

 

Analysis
  • Data Profiling (Metadata, min/max values, etc)
    • What columns have been profiled?
    • How is the profiling done? How often is it updated?
    • Are any gaps in data properly communicated to stakeholders?
  • Aggregate rules (What columns are generated via aggregation? Are there rules regarding regeneration of data due to backfills or updates?)
  • Lookup tables (What lookup tables were created for the project? What are they for?)
  • Findings (What did our analysis turn up? How did we get those results and are the reproducible?)
  • Recommendations (Any results here should be well documented as part of the Design Document linked at the start of this checklist)

 

Storage
  • Day 1 size requirements 
    • How much space needs to be budgeted for with the initial backfill of data from the source systems?
    • If cost can be calculated, we should provide it
  • Predicted growth
    • Over months? Quarters? Years?
    • How consistent do we expect the growth to be?
    • Is it sustainable to store all data for the foreseeable future? 
  • Archive rules
    • How long should data be stored before archiving?
    • Where is it archived?
    • How difficult or costly is it to retrieve archived information?
    • Are there any regulations about data that need to be considered? GDPR, for instance?
  • Disaster recovery
    • Is there a DR plan?
    • How quickly can data be recovered?
    • What changes to the process happen for older data vs newer data?
