Introduction

Datahub is a collection of Docker containers, launched with Docker-Compose, and managed by a (kludgy) CLI.  The CLI is written in Python and can be reviewed  to makeup for missing details in the documentation.  Essentially, the CLI is a wrapper around the docker-compose, docker pull and other Docker commands.  

Sandbox

The Sandbox environment can be found at https://datahub.qbiz-wiki.com.  It runs on a standard AWS t2.large instance with Docker and Docker-Compose installed.

Shell-level access

 To access the instance's Shell and Datahub CLI requires either:

  1. possession of the datahub.pem file to access the instance via SSH with, ssh -i datahub.pem ec2-user@datahub.qbiz-wiki.com.
  2. Giving a public key to the  owner of the datahub.pem key so they can install it into the ~/.ssh/authorized_keys file.  This is the preferred method for the following reasons:
    1. It grants individual access that can be removed at anytime without impacting the access of other users.
    2. No public sharing of a common key that is passed around through slack, email or other leaky communication channels.

Web Interface access

Request access from an administrator – currently Soren and Andres.  The administrator will login to the web interface and send you an invitation via e-mail.

Installation

The Datahub Quickstart Guide is the only Installation document and anything with Quickstart in the name isn't going to go into much detail.  The username and password are both datahub so you will want to change this pretty quickly if running exposed to the internet – such as our sandbox instance in AWS.

HTTPS setup

The webserver does support HTTPS, but you will need to set a variable in the configuration file to enable it and restart the container.  However, this doesn't include a Certificate Authority.  For Certificate Authority support, you have the following options:

  1. Install a Let's Encrypt client on the server and modify the Docker Compose file to mount the location were the certificates will be stored – and updated.
  2. Install a Reverse Proxy to handle the SSL communication and forward it to the datahub port.

I chose both, but the settled on the second option and Caddy as the reverse proxy.  Caddy has a Let's Encrypt client installed by default and will handle the certificate updates; all you need to do is pass it the correct  configuration file.

Caddy

Caddyfile (configuration)

datahub.qbiz-wiki.com {
        # Another common task is to set up a reverse proxy:
        reverse_proxy http://datahub.qbiz-wiki.com:9002
}

To launch a Docker Container so that Caddyfile can be seen in the container, use the following command:
cd caddy; docker run -d --restart always -p 443:443 -p80:80 -v $PWD/conf:/etc/caddy -v caddy_data:/data caddy

Lastly, edit the /etc/hosts file so that traffic won't be forwarded over the internet to reach the server from the proxy.  It should look something like this:
 

127.0.0.1   localhost localhost.localdomain localhost4 localhost4.localdomain4 datahub.qbiz-wiki.com
::1         localhost6 localhost6.localdomain6
